# 架构与设计权衡

## 1. 设计目标

| 关键因素 | 目标 | 主要手段 |
| --- | --- | --- |
| **准确度** | 中文准确率 ≥ 浏览器原生 ASR 基线，在专业领域显著优于基线 | 浏览器 ASR + 服务端文本增强 + 用户热词词库 |
| **响应速度** | 首字延迟 < 300ms，连续转写无明显停顿 | 流式 interim results、UI 直接渲染浏览器原生事件、后处理异步 |
| **易用性** | 新用户 30 秒内可完成首次有效输入 | 单按钮录音、可视化录音状态、语音命令、键盘快捷键、明暗主题 |
| **成本** | 单用户 0 元 / 月 ASR 调用费用，可零基础设施部署 | 浏览器内 Web Speech API + 后端无外部依赖 + 文件存储 |
| **隐私** | 音频数据不离开用户浏览器 | 仅文本走后端，后端可完全本地化部署 |

## 3. 关键决策与权衡

### 3.1 ASR 引擎：为什么选 Web Speech API？

| 候选方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| **Web Speech API** | 浏览器内置、零成本、低延迟、隐私友好 | Firefox 不支持；准确率不及云服务 | ✅ 主方案 |
| 云 ASR（百度/讯飞/阿里） | 准确率最高、领域适配 | 需 API Key、按时计费、网络往返 | ❌ 一期不引入，预留接口 |
| 本地 Whisper | 离线、可控 | 模型大（base ≥ 145MB）、首次加载慢、CPU 占用 | ❌ 评审环境难以演示 |
| 混合方案 | 兼顾速度与准确率 | 工程复杂、双轨调试 | ❌ 第二期再考虑 |

**取舍**：在评审 / 演示场景里，"零部署、点开就能用"压倒性优于 1–2% 的准确率提升。准确率短板用**服务端文本增强 + 用户热词**补足，把"识别准确"分解为"声学准确 + 语义/术语后处理"两个独立环节。

### 3.2 后端：为什么是 Node.js + Express + 文件存储？

- **Node.js**：与前端同语言，复用 JS 心智模型。Express 是事实标准，Express 4 系列稳定且无破坏性升级压力。
- **文件存储**：一期没有多用户、多端同步需求。引入 SQLite/Postgres 会让评审者多一步部署成本，违反"零基础设施"目标。`store.js` 隐藏了存储细节，未来切换到 Redis / SQLite 不影响路由层。
- **没有 WebSocket**：浏览器内 ASR 事件本身就是流式，文本回传后端的频率很低（每次 final），HTTP 完全够用。引入 WS 会增加状态管理复杂度。

### 3.3 文本后处理：为什么放在服务端而不是客户端？

| 维度 | 客户端 | 服务端 |
| --- | --- | --- |
| 词库更新 | 需更新前端 bundle | 单点更新，所有客户端立即受益 |
| 跨平台 | 每个平台重写 | 一次实现 |
| 算力 | 浏览器较弱 | 可升级（未来接 LLM） |
| 离线能力 | ✅ | ❌ |

一期选择服务端，是为了未来可平滑升级到 LLM 后处理（语法纠错、长句重写）而不影响前端。

### 3.4 语音命令：为什么走服务端解析？

把"删除/换行/句号"这种命令词的判断做在服务端有两个好处：
1. **可扩展**：未来要支持更多命令（"翻页""加粗"），只改后端配置。
2. **多语言**：如果同一台设备切换中英文，后端集中维护匹配规则，前端无感。

代价是每条 final 文本多一次 HTTP 请求 — 通过设计兜底逻辑（命令服务失败时直接当文本插入），把代价限制在"功能降级"而不是"功能阻塞"。

### 3.5 热词机制

热词分为两层：
1. **服务端别名替换**（已实现）：用户定义 `Kubernetes` 的别名 `k8s/K8S`，识别完成后做正则替换。
2. **未来可接 ASR 的 grammar / phrase-list 接口**（预留）：把热词作为 hint 传给 ASR 引擎，影响声学解码本身。

一期只做第 1 层，因为浏览器 Web Speech API 不开放 phrase list；上云 ASR 时再启用第 2 层。

## 4. 性能预算

| 阶段 | 预算 | 实际 (Chrome 130, 本地后端) |
| --- | --- | --- |
| 首字延迟（说出 → 屏幕） | < 300ms | ~150–250ms（浏览器原生） |
| Final → 文本入框 | < 50ms | <10ms（同步插入） |
| `/api/enhance` 往返 | < 100ms | ~5–15ms（本地） |
| `/api/command` 往返 | < 100ms | ~3–8ms（本地） |

后端每个端点都是无状态、无 I/O 阻塞（store 读小文件），在 4 核机器上 RPS 上限远高于单用户场景的需求。

## 5. 安全与隐私

- **音频不出浏览器**：Web Speech API 直接在浏览器进程内处理音频，不通过我们的服务器。
- **CORS**：后端开放 CORS 仅为方便本地双端开发，生产部署应同源或加白名单。
- **输入大小限制**：`express.json({ limit: '256kb' })` 防止恶意大 payload。
- **错误响应**：错误处理中间件不返回堆栈，避免信息泄漏。
- **存储隔离**：`backend/data/` 在 `.gitignore` 中，避免历史记录意外提交。

## 6. 演进路线

1. **阶段 1（当前）**：浏览器 ASR + 服务端文本增强 + 文件存储。
2. **阶段 2**：接入云 ASR 作为兜底，针对长音频与方言场景；切换到 SQLite。
3. **阶段 3**：引入轻量 LLM（如 Qwen2-1.5B）做语法纠错与长句润色，可选关闭以保证响应速度。
4. **阶段 4**：用户系统 + 多端同步 + WebSocket 实时协作转写。

## 7. 工程实践

- **Conventional Commits**：`feat / fix / chore / docs / test / refactor` 前缀，message 解释 *why* 而非 *what*。
- **PR 流程**：每个独立模块独立分支（`feat/backend-api`、`feat/frontend-ui`、`docs/architecture`），通过 `--no-ff` 合并保留分支拓扑。
- **测试覆盖**：核心后处理逻辑 100% 行覆盖（jest --coverage）。前端为纯 UI 编排，依赖手动验证。
- **可观测性**：前端展示首字延迟，方便用户感知系统性能；后端 `/api/health` 提供基础健康探针。
